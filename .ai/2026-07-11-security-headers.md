# Security Headers Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use $superpowers-subagent-driven-development (recommended) or $superpowers-executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Apply and verify a comprehensive browser security-header policy across prerendered pages, static assets, and dynamic Cloudflare Worker responses.

**Architecture:** Define one documented policy, use SvelteKit CSP configuration for framework-rendered HTML, and use Cloudflare response handling for headers that must cover static assets as well as Worker responses. Keep authenticated responses private and preserve the Supabase SSR refresh headers already emitted by `src/hooks.server.ts`.

**Tech Stack:** SvelteKit 2, Svelte 5, Cloudflare Workers Static Assets, TypeScript, Node test runner

## Global Constraints

- Preserve the pre-paint theme behavior in `src/app.html` without adding `unsafe-inline` to `script-src`.
- Do not cache responses that set or refresh authentication cookies.
- Enable HSTS only at the production HTTPS boundary.
- Local development must continue to support the Vite dev server and local Supabase URLs.
- Verify the final deployed headers; configuration-only evidence is insufficient.

### Task 1: Inventory Required Origins And Response Classes

**Files:**

- Create: `docs/security-headers.md`
- Inspect: `src/app.html`
- Inspect: `src/hooks.server.ts`
- Inspect: `vite.config.ts`
- Inspect: `wrangler.jsonc` once created by the deployment plan

**Interfaces:**

- Consumes: Current theme bootstrap, Supabase auth calls, fonts, SVG assets, and deployment topology
- Produces: A policy table covering directives, allowed origins, environments, and response classes

- [ ] **Step 1: Record every browser-loaded origin**

  Document `self`, the production Supabase HTTPS/WSS origins, local Vite and local Supabase origins, and whether each is required by `connect-src`, `img-src`, `font-src`, or another directive. Do not allow an origin without a concrete caller.

- [ ] **Step 2: Define response classes**

  Include prerendered HTML, immutable assets, dynamic public pages, authentication pages/actions, and learner API responses. State that any response carrying `Set-Cookie` must use `Cache-Control: private, no-store`.

- [ ] **Step 3: Define the target policy**

  Start with `default-src 'self'`, `base-uri 'self'`, `object-src 'none'`,
  `frame-ancestors 'none'`, `form-action 'self'`, `img-src 'self' data:`,
  `font-src 'self'`, and `style-src 'self' 'unsafe-inline'`. Authorize the
  pre-paint script with the hash generated by SvelteKit and add only the exact
  Supabase origins found in the inventory to `connect-src`. Add
  `upgrade-insecure-requests` to the production policy only.

  Document why `style-src 'unsafe-inline'` is temporarily required if Svelte or component behavior still emits inline styles, and create a measurable follow-up condition for removing it.

- [ ] **Step 4: Commit the policy document**

  ```bash
  git add docs/security-headers.md
  git commit -m "docs: define browser security header policy"
  ```

- [ ] **Step 1: Write failing policy tests**

  Assert that the production policy contains `default-src 'self'`, `object-src 'none'`, `base-uri 'self'`, `frame-ancestors 'none'`, `form-action 'self'`, and `upgrade-insecure-requests`; assert that development does not upgrade local HTTP requests. Assert that authenticated responses become `private, no-store`.

- [ ] **Step 2: Run the focused tests and confirm failure**

  ```bash
  node --test src/lib/server/security-headers.test.ts
  ```

  Expected: failure because the helper module does not exist.

- [ ] **Step 3: Implement the helpers**

  Return a cloned response with `Content-Security-Policy`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-Content-Type-Options: nosniff`, and `Permissions-Policy: camera=(), microphone=(), geolocation=()` set. Use CSP `frame-ancestors` as the primary framing protection and add `X-Frame-Options: DENY` for legacy clients.

- [ ] **Step 4: Apply the helper after `resolve`**

  Refactor `src/hooks.server.ts` so it awaits `resolve`, preserves Supabase-provided cache headers from the `setAll` callback, and applies the common policy without overwriting `private, no-store`.

- [ ] **Step 1: Add static response headers**
