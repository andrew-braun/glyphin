# Docs Rules

## Purpose

- `docs/` contains durable reference documentation for the repository.
- Documents here should remain useful after the original task is complete.

## What Belongs Here

- Security checklists
- Architecture and workflow guides
- Long-lived operational references
- Durable decision records when a separate ADR system is not in use
- Database reference docs such as `db.md` and `database-dto-spec.md`

## What Does Not Belong Here

**Hard rule: no planning documents in `docs/`. Ever. They all go in `.ai/`.**

- Implementation plans and design specs — including any generated by an external tool, skill, or workflow. A tool's default output path is never a reason to leave a plan here; move it to `.ai/` in the same change.
- Transient task trackers, rollout plans, launch checklists, and audit follow-ups
- One-off scratch notes
- Chat-history summaries that only matter to a single implementation pass
- Nested plan directories of any name (`docs/plans/`, `docs/specs/`, tool-named subfolders). A `docs/superpowers/` tree accumulated eight plan files before the 2026-07-14 audit caught it. Do not recreate it.

The test: **if it has checkboxes, a status line, or a "phases"/"steps" section, it is a plan — it belongs in `.ai/`.** If it describes how the system _is_ rather than what someone intends to _do_, it belongs here.

## Maintenance

- Keep docs concise, factual, and easy to scan.
- Update docs when the implementation or policy they describe changes.
- Delete or rewrite stale guidance immediately.
- Keep `db.md` as the README-style entry point for database layout and inspection, and keep `database-dto-spec.md` as the exact schema contract.
